# Business Rules - Unit 3: order-sse

## 1. 주문 번호 생성 규칙

### BR-ORD-001: 날짜 기반 주문번호 생성
- **형식**: `YYYYMMDD-NNN` (예: 20260318-001)
- **NNN**: 매장별 일일 시퀀스 (001부터 시작, 매일 리셋)
- **고유성**: storeId + orderNumber 조합으로 유니크
- **생성 시점**: 주문 생성 트랜잭션 내에서 원자적으로 생성
- **동시성**: 같은 매장에서 동시 주문 시 시퀀스 충돌 방지 (DB 레벨 락 또는 유니크 제약)

---

## 2. 주문 생성 규칙

### BR-ORD-002: 주문 생성 전제조건
- 유효한 테이블 세션이 존재해야 함 (sessionId 검증)
- 장바구니에 1개 이상의 항목이 있어야 함
- 모든 메뉴 ID가 해당 매장에 존재하고 활성 상태여야 함
- 고객 JWT 토큰이 유효해야 함

### BR-ORD-003: 주문 항목 스냅샷
- 주문 생성 시 메뉴명, 단가를 OrderItem에 스냅샷으로 저장
- 이후 메뉴 가격이 변경되어도 기존 주문 금액에 영향 없음

### BR-ORD-004: 총 금액 계산
- `totalAmount = SUM(각 OrderItem의 price × quantity)`
- 서버에서 재계산 (클라이언트 전송 금액은 참고용, 서버 계산이 최종)

### BR-ORD-005: 주문 성공 후 처리
- 주문 번호 반환
- SSE로 `new-order` 이벤트 브로드캐스트 (해당 매장)
- 클라이언트: 장바구니 비우기 → 5초 후 메뉴 화면 리다이렉트

---

## 3. 주문 상태 변경 규칙

### BR-ORD-006: 순방향 전이만 허용
- PENDING → PREPARING (허용)
- PREPARING → COMPLETED (허용)
- PENDING → COMPLETED (불허 - 단계 건너뛰기 금지)
- PREPARING → PENDING (불허 - 역방향 금지)
- COMPLETED → * (불허 - 최종 상태)

### BR-ORD-007: 상태 변경 권한
- 관리자만 상태 변경 가능 (관리자 JWT 필수)
- 상태 변경 시 SSE로 `order-status` 이벤트 브로드캐스트

### BR-ORD-008: 유효하지 않은 상태 전이 시
- HTTP 400 에러 반환
- 현재 상태와 요청 상태를 에러 메시지에 포함

---

## 4. 주문 삭제 규칙

### BR-ORD-009: 주문 삭제 권한
- 관리자만 삭제 가능
- 모든 상태의 주문 삭제 가능 (PENDING, PREPARING, COMPLETED)

### BR-ORD-010: 삭제 후 처리
- 해당 주문의 OrderItem도 함께 삭제 (cascade)
- 테이블 총 주문액 재계산
- SSE로 `order-deleted` 이벤트 브로드캐스트 (재계산된 총액 포함)

---

## 5. 주문 조회 규칙

### BR-ORD-011: 고객 주문 조회
- 현재 테이블 세션(sessionId)의 주문만 조회 가능
- 주문 시간 순 정렬 (최신 먼저)
- 이용 완료 처리된 주문은 제외 (OrderHistory로 이동된 것)

### BR-ORD-012: 관리자 주문 조회
- 매장 전체 활성 주문 조회 (테이블별 그룹)
- 각 테이블: 총 주문액, 최신 3개 주문 미리보기
- 주문 카드 클릭 시 전체 메뉴 목록 상세 보기

### BR-ORD-013: 과거 주문 내역 조회
- OrderHistory 테이블에서 조회
- 테이블별, 시간 역순 정렬
- 날짜 필터링 지원

---

## 6. 장바구니 규칙 (클라이언트)

### BR-CART-001: 장바구니 저장
- localStorage에 저장 (키: `cart_{storeId}_{tableId}`)
- 새로고침 시 유지
- 서버 전송은 주문 확정 시에만

### BR-CART-002: 수량 관리
- 최소 수량: 1
- 수량 0이 되면 해당 항목 자동 삭제
- 동일 메뉴 추가 시 수량 자동 증가

### BR-CART-003: 총 금액 계산
- 실시간 계산: `SUM(각 항목의 price × quantity)`
- 화면에 즉시 반영

### BR-CART-004: 빈 장바구니 주문 방지
- 장바구니가 비어있으면 주문 확정 버튼 비활성화

---

## 7. SSE 규칙

### BR-SSE-001: 연결 관리
- 매장별 클라이언트 풀 관리 (Map<storeId, Set<SSEClient>>)
- 연결 시 `Content-Type: text/event-stream` 헤더 설정
- keep-alive: 30초 간격 heartbeat 전송

### BR-SSE-002: 이벤트 브로드캐스트
- 해당 매장의 모든 연결된 클라이언트에게 전송
- 이벤트 형식: `event: {eventName}\ndata: {JSON}\n\n`
- 전송 실패 시 해당 클라이언트 연결 제거

### BR-SSE-003: 클라이언트 재연결
- 즉시 재연결 시도 + 지수 백오프 (1초, 2초, 4초... 최대 30초)
- 최대 재시도 횟수: 10회 (약 2분)
- 10회 초과 시: 재연결 중단 → "연결 끊김" 상태 표시 + "다시 연결" 수동 버튼 제공
- 수동 버튼 클릭 시 재시도 카운터 리셋 후 재연결 시작
- 연결 상태 UI 표시 (연결됨/재연결 중/연결 끊김)
- 재연결 성공 시 최신 데이터 전체 새로고침, 재시도 카운터 리셋

### BR-SSE-004: 2초 이내 전달
- 주문 생성 → SSE 이벤트 전달까지 2초 이내 (NFR 요구사항)
- 브로드캐스트는 비동기 처리 (주문 응답 지연 방지)

---

## 8. 데이터 무결성 규칙

### BR-INT-001: 주문 생성 트랜잭션
- Order + OrderItem 생성은 단일 트랜잭션으로 처리
- 실패 시 전체 롤백

### BR-INT-002: 세션 기반 주문 격리
- 주문은 반드시 유효한 세션에 속해야 함
- 세션 종료 후 해당 세션의 주문 생성 불가

### BR-INT-003: 멀티 테넌트 격리
- 모든 주문 API는 storeId로 필터링
- 다른 매장의 주문 접근 불가

## 9. Unit 1 (core-auth) 협의 필요 사항

> ⚠️ 아래 항목은 Unit 1 개발 완료 시 담당자와 협의 후 확정 필요

### PENDING-001: JWT Payload 구조 확정
- Unit 3은 JWT payload에 `storeId`, `tableId`, `sessionId`, `role` 필드가 포함된다고 가정
- Unit 1에서 실제 payload 구조가 확정되면 Unit 3의 토큰 파싱 로직 조정 필요

### PENDING-002: authMiddleware 인터페이스
- Unit 3은 `authMiddleware`가 디코딩된 payload를 `req.user`에 설정한다고 가정
- 실제 미들웨어 구현 패턴 확인 후 라우트 핸들러 조정 필요

### PENDING-003: 고객/관리자 역할 구분 방식
- Unit 1의 Functional Design Q2 (JWT 토큰 구분 방식)가 미답변 상태
- `role` 필드 기반 구분을 가정하고 있으나, 별도 시크릿 키 방식이 채택되면 미들웨어 사용 방식 변경 필요

### PENDING-004: 세션 만료 시 주문 API 동작
- 토큰 만료 시 401 반환 → 클라이언트에서 자동 재로그인 시도 (Unit 1 authStore 담당)
- 재로그인 중 장바구니 데이터는 localStorage에 유지됨 (Unit 3 cartStore)
