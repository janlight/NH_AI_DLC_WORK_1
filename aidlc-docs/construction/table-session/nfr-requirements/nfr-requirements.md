# NFR Requirements - Unit 4: table-session

## NFR-T01: 데이터 무결성 - 이용 완료 트랜잭션

**요구사항**: 이용 완료 처리는 원자적 트랜잭션으로 실행되어야 함
**기준**:
- Prisma `$transaction` 사용, 타임아웃 60초
- 트랜잭션 내 모든 단계 실패 시 전체 롤백
- 부분 완료 상태 발생 불가

**검증 방법**: 트랜잭션 중간 실패 시나리오 테스트 (주문 이력 이동 후 삭제 실패 시 롤백 확인)

## NFR-T02: 동시성 제어 - 낙관적 잠금

**요구사항**: 이용 완료와 주문 생성 동시 발생 시 데이터 정합성 보장
**기준**:
- TableSession에 `version` 필드 추가 (Int, default: 0)
- 이용 완료 시 version 체크 후 업데이트 (`WHERE version = currentVersion`)
- 충돌 시 `PrismaClientKnownRequestError` (P2025) 감지 → 409 Conflict 응답
- 클라이언트에서 재시도 안내

**검증 방법**: 동시 요청 시뮬레이션 테스트

## NFR-T03: 데이터 보존 - 과거 주문 내역

**요구사항**: 90일 보존 후 자동 삭제
**기준**:
- OrderHistory.expiresAt = 생성 시각 + 90일
- 일일 배치 작업 (cron job)으로 만료 레코드 삭제
- 배치 실행: 매일 03:00 (새벽, 트래픽 최소 시간대)
- 삭제 시 배치 크기 제한 (1000건씩) - DB 부하 방지

**검증 방법**: expiresAt 기준 삭제 동작 확인, 배치 로그 검증

## NFR-T04: 성능 - 테이블 관리 페이지

**요구사항**: 매장당 100개 테이블까지 1초 이내 로딩
**기준**:
- `GET /api/stores/:storeId/tables` 응답 시간 < 1초 (100개 테이블)
- 테이블별 현재 세션 주문 총액을 DB 쿼리 레벨에서 집계 (N+1 방지)
- 인덱스: `Table(storeId)`, `Order(tableId, sessionId)`

**검증 방법**: 100개 테이블 + 각 10건 주문 데이터로 응답 시간 측정

## NFR-T05: 보안 - 매장 데이터 격리

**요구사항**: 멀티 테넌트 환경에서 매장 간 데이터 접근 차단
**기준**:
- 모든 API에서 JWT의 storeId와 요청 경로의 storeId 일치 검증
- 테이블/세션/주문 조회 시 항상 storeId 조건 포함
- 다른 매장의 테이블 이용 완료 처리 불가

**검증 방법**: 다른 매장 storeId로 API 호출 시 403 응답 확인

## NFR-T06: 신뢰성 - SSE 이벤트 발행

**요구사항**: 이용 완료/주문 삭제 이벤트가 관리자 대시보드에 확실히 전달
**기준**:
- 트랜잭션 커밋 후 SSE 이벤트 발행 (트랜잭션 내부 아님)
- SSE 발행 실패 시 이용 완료 트랜잭션은 롤백하지 않음 (best-effort)
- 관리자 대시보드 주기적 폴링 fallback (30초 간격)

**검증 방법**: SSE 연결 끊김 상태에서 이용 완료 후 재연결 시 최신 상태 반영 확인
