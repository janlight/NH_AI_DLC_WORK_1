# 사용자 스토리

## EP-01: 테이블 인증 및 세션 관리 (고객)

### US-01-01: 테이블 태블릿 초기 로그인
**As a** 테이블 고객 (P1, P2),
**I want to** 관리자가 설정한 테이블에서 별도 로그인 없이 주문 화면에 접근하고 싶다,
**So that** 대기 시간 없이 바로 메뉴를 탐색하고 주문할 수 있다.

**수용 기준:**
- [ ] 저장된 로그인 정보(매장 ID, 테이블 번호, 비밀번호)로 자동 로그인 수행
- [ ] 자동 로그인 성공 시 메뉴 화면으로 이동
- [ ] 자동 로그인 실패 시 수동 로그인 화면 표시 (매장 식별자, 테이블 번호, 비밀번호 입력)
- [ ] 로그인 성공 시 인증 정보 로컬 저장
- [ ] JWT 토큰 발급 및 16시간 세션 유지
- [ ] 브라우저 새로고침 시 세션 유지
- [ ] 터치 친화적 입력 필드 (최소 44x44px)

### US-01-02: 세션 만료 처리
**As a** 테이블 고객 (P1, P2),
**I want to** 세션이 만료되면 자동으로 재로그인되길 원한다,
**So that** 장시간 이용 시에도 끊김 없이 서비스를 사용할 수 있다.

**수용 기준:**
- [ ] 16시간 세션 만료 시 저장된 정보로 자동 재로그인 시도
- [ ] 재로그인 실패 시 수동 로그인 화면으로 이동
- [ ] 세션 만료 중 진행 중이던 장바구니 데이터 유지 (로컬 저장)

## EP-02: 메뉴 조회 및 탐색 (고객)

### US-02-01: 카테고리별 메뉴 조회
**As a** 테이블 고객 (P1, P2),
**I want to** 카테고리별로 분류된 메뉴를 탐색하고 싶다,
**So that** 원하는 메뉴를 빠르게 찾을 수 있다.

**수용 기준:**
- [ ] 로그인 후 메뉴 화면이 기본 화면으로 표시
- [ ] 카테고리 목록 표시 및 카테고리 간 빠른 이동
- [ ] 카드 형태 메뉴 레이아웃
- [ ] 각 메뉴 카드에 메뉴명, 가격, 이미지 표시
- [ ] 터치 친화적 버튼 크기 (최소 44x44px)
- [ ] 명확한 시각적 계층 구조
- [ ] 메뉴 데이터 로딩 실패 시 에러 메시지 표시 및 재시도 버튼

### US-02-02: 메뉴 상세 정보 확인
**As a** 테이블 고객 (P1, P2),
**I want to** 메뉴의 상세 정보를 확인하고 싶다,
**So that** 주문 전에 메뉴에 대해 충분히 파악할 수 있다.

**수용 기준:**
- [ ] 메뉴 카드 클릭 시 상세 정보 표시 (메뉴명, 가격, 설명, 이미지)
- [ ] 상세 화면에서 장바구니 추가 가능
- [ ] 닫기/뒤로가기로 메뉴 목록 복귀

## EP-03: 장바구니 관리 (고객)

### US-03-01: 장바구니에 메뉴 추가
**As a** 테이블 고객 (P1, P2),
**I want to** 원하는 메뉴를 장바구니에 담고 싶다,
**So that** 여러 메뉴를 모아서 한 번에 주문할 수 있다.

**수용 기준:**
- [ ] 메뉴 카드 또는 상세 화면에서 장바구니 추가 버튼
- [ ] 추가 시 시각적 피드백 (애니메이션 또는 알림)
- [ ] 장바구니 아이콘에 담긴 수량 표시
- [ ] 동일 메뉴 추가 시 수량 자동 증가

### US-03-02: 장바구니 수정 및 관리
**As a** 테이블 고객 (P1, P2),
**I want to** 장바구니의 메뉴 수량을 조절하거나 삭제하고 싶다,
**So that** 주문 전에 원하는 대로 조정할 수 있다.

**수용 기준:**
- [ ] 수량 증가/감소 버튼
- [ ] 수량 0 시 해당 메뉴 자동 삭제 또는 삭제 확인
- [ ] 개별 메뉴 삭제 버튼
- [ ] 장바구니 전체 비우기 버튼
- [ ] 총 금액 실시간 계산 및 표시
- [ ] 클라이언트 측 로컬 저장 (새로고침 시 유지)
- [ ] 빈 장바구니 상태 안내 메시지

## EP-04: 주문 생성 및 내역 조회 (고객)

### US-04-01: 주문 확정
**As a** 테이블 고객 (P1, P2),
**I want to** 장바구니의 메뉴를 주문으로 확정하고 싶다,
**So that** 매장에서 음식을 준비받을 수 있다.

**수용 기준:**
- [ ] 주문 내역 최종 확인 화면 (메뉴 목록, 수량, 단가, 총 금액)
- [ ] 주문 확정 버튼
- [ ] 성공 시: 주문 번호 표시 → 장바구니 자동 비우기 → 5초 후 메뉴 화면 리다이렉트
- [ ] 실패 시: 에러 메시지 표시, 장바구니 유지
- [ ] 주문 정보에 매장 ID, 테이블 ID, 세션 ID 포함
- [ ] 빈 장바구니로 주문 시도 시 방지

### US-04-02: 주문 내역 조회
**As a** 테이블 고객 (P1, P2),
**I want to** 현재 테이블의 주문 내역을 확인하고 싶다,
**So that** 이전에 주문한 내용과 상태를 파악할 수 있다.

**수용 기준:**
- [ ] 현재 테이블 세션의 주문만 표시 (이전 세션 제외)
- [ ] 주문 시간 순 정렬
- [ ] 주문별 정보: 주문 번호, 시각, 메뉴/수량, 금액, 상태(대기중/준비중/완료)
- [ ] 이용 완료 처리된 주문 제외
- [ ] 주문 없을 시 안내 메시지

## EP-09: 비기능 요구사항

### US-09-01: 보안
**As a** 시스템 운영자,
**I want to** 시스템이 보안 기준을 충족하길 원한다,
**So that** 사용자 데이터와 시스템이 안전하게 보호된다.

**수용 기준:**
- [ ] 비밀번호 bcrypt 해싱 저장
- [ ] JWT 토큰 기반 인증 (고객/관리자)
- [ ] 로그인 시도 제한
- [ ] 데이터 전송 시 암호화 (HTTPS)
- [ ] SQL Injection, XSS 방지

### US-09-02: 성능
**As a** 시스템 운영자,
**I want to** 시스템이 빠른 응답 속도를 제공하길 원한다,
**So that** 사용자가 쾌적하게 서비스를 이용할 수 있다.

**수용 기준:**
- [ ] SSE 기반 실시간 주문 알림 2초 이내
- [ ] 메뉴 조회 응답 시간 최소화
- [ ] 동시 접속 처리 (매장당 30~100 테이블)

### US-09-03: 확장성
**As a** 시스템 운영자,
**I want to** 시스템이 다수 매장을 지원하길 원한다,
**So that** SaaS 형태로 100개 이상 매장에 서비스를 제공할 수 있다.

**수용 기준:**
- [ ] 멀티 테넌트 아키텍처 (매장별 데이터 격리)
- [ ] 100개+ 매장 지원 가능한 DB 설계
- [ ] Docker Compose 기반 배포 환경

### US-09-04: 데이터 무결성
**As a** 시스템 운영자,
**I want to** 데이터의 일관성과 영속성이 보장되길 원한다,
**So that** 주문 데이터 손실이나 불일치가 발생하지 않는다.

**수용 기준:**
- [ ] 세션 기반 주문 격리
- [ ] 주문 이력 영구 보존 (OrderHistory)
- [ ] 장바구니 로컬 저장 (새로고침 유지)
- [ ] 트랜잭션 기반 주문 처리

## INVEST 기준 검증 요약

| 기준 | 충족 방법 |
|---|---|
| **Independent** | 각 스토리가 독립적으로 개발/테스트 가능하도록 에픽 내에서 분리 |
| **Negotiable** | 수용 기준은 체크리스트 형식으로 협상 가능 |
| **Valuable** | 각 스토리가 고객 또는 관리자에게 직접적 가치 제공 |
| **Estimable** | 중간 세분화 수준으로 추정 가능한 크기 |
| **Small** | 주요 사용자 액션 단위로 적절한 크기 유지 |
| **Testable** | 체크리스트 형식 수용 기준으로 테스트 가능 |
