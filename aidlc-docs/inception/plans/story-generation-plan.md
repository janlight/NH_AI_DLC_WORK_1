## 스토리 생성 방법론

### 접근 방식 옵션

| 접근 방식 | 설명 | 장점 | 단점 |
|---|---|---|---|
| **User Journey-Based** | 사용자 워크플로우 순서대로 스토리 구성 | 사용자 경험 흐름 파악 용이 | 공통 기능 중복 가능 |
| **Feature-Based** | 시스템 기능 단위로 스토리 구성 | 개발 단위와 매핑 용이 | 사용자 관점 약화 가능 |
| **Persona-Based** | 사용자 유형별로 스토리 그룹화 | 페르소나별 니즈 명확 | 공통 기능 분리 어려움 |
| **Epic-Based** | 대규모 에픽 → 하위 스토리 계층 구조 | 범위 관리 용이 | 초기 구조화 비용 |
| **Hybrid** | 에픽(기능 기반) + 내부 스토리(사용자 여정 기반) | 균형 잡힌 접근 | 규칙 정의 필요 |

## 질문 (Step 3)

### Q1: 스토리 분류 접근 방식
위 표의 접근 방식 중 어떤 방식을 선호하시나요?

- A) User Journey-Based - 사용자 워크플로우 순서대로
- B) Feature-Based - 시스템 기능 단위로
- C) Persona-Based - 사용자 유형별로
- D) Epic-Based - 에픽 → 하위 스토리 계층 구조
- E) Hybrid - 에픽(기능 기반) + 내부 스토리(사용자 여정 기반) (권장)

[Answer]: E

### Q2: 스토리 세분화 수준
각 스토리의 크기를 어느 정도로 세분화할까요?

- A) 대규모 - 기능 단위 (예: "메뉴 관리" 하나의 스토리)
- B) 중간 - 주요 사용자 액션 단위 (예: "메뉴 등록", "메뉴 수정" 각각 스토리) (권장)
- C) 세밀 - 개별 인터랙션 단위 (예: "메뉴 이미지 업로드", "메뉴명 입력" 각각 스토리)

[Answer]: B

### Q3: 수용 기준(Acceptance Criteria) 형식
각 스토리의 수용 기준을 어떤 형식으로 작성할까요?

- A) Given-When-Then (BDD 스타일) - 구조적이고 테스트 자동화에 유리
- B) 체크리스트 형식 - 간결하고 빠른 검증 가능 (권장)
- C) 자유 서술형 - 유연하지만 일관성 낮음

[Answer]: B

### Q4: 고객 페르소나 세분화
고객(테이블 이용자) 페르소나를 어떻게 정의할까요?

- A) 단일 페르소나 - "테이블 고객" 하나로 통합 (권장 - MVP 단계)
- B) 다중 페르소나 - 연령대/디지털 친숙도별 구분 (예: 젊은 고객, 시니어 고객)

[Answer]: B

### Q5: 관리자 페르소나 세분화
관리자 페르소나를 어떻게 정의할까요?

- A) 단일 페르소나 - "매장 관리자" 하나로 통합 (권장 - MVP 단계)
- B) 다중 페르소나 - 역할별 구분 (예: 매장 오너, 홀 매니저)

[Answer]: B

### Q6: 비기능 요구사항 스토리 포함 여부
보안, 성능 등 비기능 요구사항도 별도 스토리로 작성할까요?

- A) 포함 - 비기능 요구사항도 별도 스토리로 작성 (권장)
- B) 제외 - 비기능 요구사항은 각 기능 스토리의 수용 기준에만 포함
- C) 부분 포함 - 보안 관련만 별도 스토리, 나머지는 수용 기준에 포함

[Answer]: A

### Q7: 에러/예외 시나리오 처리
에러 및 예외 상황을 스토리에 어떻게 반영할까요?

- A) 별도 스토리 - 주요 에러 시나리오를 독립 스토리로 작성
- B) 수용 기준에 포함 - 각 기능 스토리의 수용 기준에 에러 케이스 포함 (권장)
- C) 혼합 - 치명적 에러는 별도 스토리, 일반 에러는 수용 기준에 포함

[Answer]: B
